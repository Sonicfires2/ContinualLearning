# Dataset: Split CIFAR-100

## Purpose

This document records the dataset facts and project-specific Split CIFAR-100 choices used by the core continual-learning study. It exists so later baseline, replay, signal, and predictor results can be interpreted against the same benchmark contract.

The raw inspection script is [scripts/inspect_cifar100.py](../scripts/inspect_cifar100.py).

## Dataset Identity

- Dataset: CIFAR-100
- Domain: small natural images
- Image shape: `32 x 32 x 3`
- Raw row shape: `3072` values, stored as channel-first flattened RGB data in the CIFAR Python files
- Fine classes: `100`
- Coarse classes: `20`
- Train examples: `50000`
- Test examples: `10000`
- Train examples per fine class: `500`
- Test examples per fine class: `100`
- Train examples per coarse class: `2500`
- Test examples per coarse class: `500`

The local raw files are expected at:

```text
data/cifar-100-python/
  meta
  train
  test
```

These files are dataset artifacts and should not be committed.

## Project Benchmark

The locked core study uses Split CIFAR-100 as:

- task count: `10`
- classes per task: `10`
- split seed: `0`
- evaluation schedule: evaluate all seen tasks after each task
- primary target for current baselines: `original_class_id`
- required metrics: final accuracy and average forgetting

Using `original_class_id` makes the current setup a class-incremental evaluation. This is intentionally stricter than task-local labels and better aligned with measuring catastrophic forgetting across the full 100-class output space.

## Stable Sample IDs

Stable sample identity is required for future signal logging, forgetting labels, replay-buffer metadata, time-to-forgetting `T_i` targets, and spaced replay analysis.

Current convention:

- train sample ID: `original_index`
- test sample ID: `1000000 + original_index`

Therefore:

- train sample IDs range from `0` to `49999`
- test sample IDs range from `1000000` to `1009999`

Do not change these offsets once sample-level signal logs exist unless a new protocol/config version is created.

## Current Preprocessing

The raw viewer reads unnormalized images directly from CIFAR-100 pickle files.

The current real baseline path uses torchvision transforms:

```text
ToTensor()
Normalize(mean=(0.5071, 0.4867, 0.4408), std=(0.2675, 0.2565, 0.2761))
```

If future experiments change preprocessing or augmentation, they must record that change in the run config and rerun all compared methods under the same preprocessing.

## Research-Relevant Caveats

- CIFAR-100 is balanced by fine class, so large performance differences should not be explained by class-count imbalance.
- The current class order is seeded, not semantically grouped. This helps avoid hand-picking an easy or hard task order.
- The first fine-tuning baseline used a lightweight `flatten_mlp`, not a strong vision architecture. It is sufficient to confirm catastrophic forgetting and validate the pipeline, but stronger architecture choices should be frozen before final comparisons.
- Random replay, fixed-periodic replay, and spaced replay must use the same task order, target key, model, optimizer, replay budget, and evaluation schedule.
- The eventual spaced scheduler must use sample IDs to connect estimated forgetting time `T_i` to the next scheduled replay step.
- Future sample-level forgetting labels must use the stable sample IDs described above.

## Current Baseline Context

The first no-replay Split CIFAR-100 run produced:

- final accuracy: `0.0455`
- average forgetting: `0.434`
- artifact directory: `experiments/runs/fine_tuning/fine_tuning_split_cifar100_seed0`

This confirms that the current benchmark setup induces severe forgetting. The next scientifically necessary comparisons are replay baselines under fixed memory and replay budgets, followed by a scheduler that estimates or approximates sample-specific forgetting time.
